## 1. Problem Statement (PS)
During peak travel seasons (festivals, vacations, academic rushes), direct train availability between major source and destination hubs experiences absolute saturation. Passengers are met with massive waitlists, yet millions of intermediate berths on those exact same routes run completely empty between non-hub stations. 

Current railway booking systems operate on linear, direct query architectures. They fail to dynamically pool and suggest staggered multi-hop availability, resulting in artificial seat scarcities for travelers and lost operational efficiency for transit systems.

---

## 2. The Solution: YatraRoute AI
YatraRoute AI treats the entire national railway schedule as a dynamic, time-weighted **Directed Acyclic Graph (DAG)**. When a direct seat from Station A to Station B is unavailable, our edge-driven in-memory algorithm processes real-time inventory schemas to automatically construct two highly efficient alternatives:

1.  **The Station Split (Multi-Hop Journey)**: Routes passengers from $A \rightarrow C$ on Train 1, provides a safety layover buffer window at junction Station C, and transitions them from $C \rightarrow B$ on Train 2.
2.  **The Seat Split (Single-Train Shifting)**: Books the **same train** from $A \rightarrow B$, but optimizes internal inventory by transitioning the passenger from Seat X (Coach B1) to Seat Y (Coach B3) at intermediate Station C, requiring zero train-hopping.

---

## 3. Uniqueness & Innovation
*   **Platform-Aware Layover Optimization**: Unlike basic transit aggregators that just look at times, YatraRoute cross-references platform and station layout maps. It prioritizes connecting trains arriving on the same or adjacent platforms to minimize heavy luggage transfer friction for families and elderly passengers.
*   **The Single-Train Seat Switcher**: Finding zero total availability but booking a modular journey inside the same train bypasses station-switching fatigue entirely. It guarantees confirmed berths through quota combining.
*   **Edge-Driven Dynamic Inventory Graph**: Instead of hammering an external backend with heavy relational searches, the algorithm runs completely in-memory on the client, resolving complex graph connections instantly.

## 5. Scalability Strategy
To demonstrate readiness for massive real-world deployment across systems like Indian Railways:
*   **Microservices Decoupling**: The **Graph Search Engine** and the **User-Facing Live Inventory Manager** are completely decoupled. The graph structure changes slowly (schedules are updated semi-annually), so transit route pathways are cached in **Redis**, cutting database read loads by over 90%.
*   **Localized Sharding**: Instead of running an algorithm over the entire global or national network simultaneously, graphs are sharded by zone or high-traffic corridors (e.g., Western Zone, Central Zone) to lower computational complexity.
*   **OTA API Partnerships**: Developed to integrate directly as a high-margin add-on API for major booking applications (RedRail, ixigo, MakeMyTrip).

## 6. How to Run Locally

### Prerequisites
*   **Node.js** (v18+)
*   **npm**

### Step 1: Install Dependencies
Install Vite, React 19, Tailwind CSS v4, and the Gemini API client:
```bash
npm install
```

### Step 2: Configure API Key
Create a `.env.local` file in the root directory (already done locally) and add your Gemini API Key:
```text
GEMINI_API_KEY=your_gemini_api_key_here
```
*(Note: If the key is not present, the app will seamlessly fall back to local rule-based text enrichment without failing.)*

### Step 3: Run the App
Start the local development server:
```bash
npm run dev
```
Open your browser and navigate to **http://localhost:3000**.
